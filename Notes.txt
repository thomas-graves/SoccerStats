Clarifying questions

A. Event taxonomy and semantics

Q1) For pass-like actions, do you want:
- pass as the main event type, with cross, through_ball, and long_ball as qualifiers
- or do you want cross, through_ball, and long_ball to remain their own top-level event types

A1) I think having pass as the main event type, with cross, through_ball and long_ball as qualifiers, is a good idea. Though I would be interested to know what you recommend? It is important to remember that their is a difference between a long pass and a long ball. There is also a difference between a long ball and a clearance.

Q2) For duels, do you want:
- one duel event type with qualifier aerial or ground
- or separate event types aerial_duel and ground_duel

A2) I think having one duel event type, with qualifiers aerial or ground is a good idea. Though I would be interested to know what you recommend. It is important to remember that a ground duel is a 1v1 battle where the ball is on the ground, there are offensive ground duels (e.g. dribbles) and defensive ground duels (e.g. tackle). A successful dribble/unsuccessful tackle might result in a won ground duel for the attacker and a lost ground duel for the defender. An unsuccessful dribble/successful tackle might result in a lost ground duel for the attacker and a won ground duel for the defender.

Q3) For cards, do you want:
- one card event type with qualifier yellow, red, or second_yellow_red
- or separate top-level event types for each card type

A3) I think one event type for card, and qualifiers for yellow, red, or second_yellow_red, is a good idea. Though again, I am interested to hear your recommendation.

Q4) For restarts, do you want:
- one restart_taken event type with qualifier like corner, free_kick, throw_in, goal_kick, penalty
- or separate top-level event types such as corner_taken

A4) Again, I think a restart_taken event type, with qualifiers like corner, free_kick, throw_in, goal_kick, and penalty is a good idea. Though it is important to remember that a free-kick might result in a shot on goal, or a pass (so will these be linked events?), a corner could be a short pass, or a cross, or a long pass, it could be successful or unsuccessful, and could result in a shot, which results in a goal (will these also be linked events? and will we be able to track goals scored from corners? or assists from corners? or number of short corners and long corners? also right hand or left hand corners?) A throw in can also be a form of pass, it could be successful or unsuccessful, it could be progressive or not (will these things be accounted for?). Similarly, a goal_kick might have different takers, not necessarily the goal keeper, it could be a short pass, or long pass, it could be grounded or lofted, it could be successful or unsuccessful (will the goal-kick event be linked to a pass event so that the type of goal_kick can be accurately recorded? or is there a better system?). Penalties will also require there own qualifiers of course.

Q5) For goals, do you want:
- goal as its own event type
- or shot with outcome scored

A5) I am not sure what is best. Is a goal event type best, which links to a shot event type? or a successful shot with outcome scored? I would be interested to know your recommendation here. Regardless, I'd like for the methodology to be consistent, let's not implement it a certain way for an event like a shot and goal, but then a different way for a thing like a pass.

Q6) For own goals, do you want:
- own_goal as its own event type
- or a normal shot/goal structure with a qualifier saying it was an own goal and who benefited

A6) I think an own_goal event would be good, as the shot would likely be recorded as an unsuccessful shot event. This system would also account for a poor pass that results in an own goal, rather than only own goals that occur from shots. That said, would this then be inconsistent if no event were to be created for a goal? I would like to know your recommendation?

Q7) For assist, key_pass, and chance_created, do you want them to be:
- derived only from linked events
- manually enterable as event types too
- or manually enterable only in rare correction cases

A7) I agree with your recommendation that assist, key_pass and chance_created should be derived, rather than event types.

- Successful tackle (Ground Duel won) by Player A, may result in an Unsuccessful Dribble and Ground Duel lost for Player B

G. Admin workflow and usability

Q23) When entering events post match, what is the expected admin workflow?
- enter full timeline minute by minute from notes/video
- enter just selected events of interest
- sometimes detailed, sometimes lightweight depending on the match

A23) It will depend on whether there is access to notes/video. If there is, then the admin will be able to enter a full timeline minute by minute using the resource. When there are no notes or video, the admin should be able to input just selected events (and there should be a note on the Matches page stating that full in-depth statistics are not available for this Match). Perhaps when the admin edits the Match Fixture to add the Match Result, they can select whether a full timeline of match events will be added or whether just a select number of key events will be included (goals / assists, cards, substitutions). When it is selected that only key events will be inputted, some statistics will still be able to be derived, but there should be a way for the admin to optionally input statistics for each player (like total shots, total shots on target, total shots off target, total blocks, corners taken, etc.), as these may have been tallied at the game by a coaching staff member. Please let me know if what I have described makes sense.

Q24) For your club’s real-world use, do you expect admins to log:
- every pass and duel
- only key passes, shots, defensive actions, cards, substitutions, and notable ball progression
- something in between

A24) As I said in my answer to Q23, it will likely be one way or the other. If the admin has access to video footage and will log every match event into the timeline. If the admin does not have access to video footage, then only key match events will be inputted, and player statistics from the game should be able to be optional inputted, as described above.


Q25) Would you like the event entry UI to eventually support:

as task-specific forms, even if they all map to the same underlying event tables?

A25) Yes, I imagine that admins will manage event entry through a user friendly UI, rather than the Django Administration, with UI that supports the entry of match events, like "add shot", "add pass", "add substitution", "begin match", "start extra time", "end half", "end match", etc. even if they all map to the same underlying event tables. This should make the manual entry much easier than one generic form.
